## Purpose（目的）

本仕様書は、**楽々精算・楽々勤怠のデータを突合し、出張精算の一次承認・二次承認に必要なチェック観点を一元化したチェックシートを作成する**ためのタスク仕様である。

実装前に、以下を関係者間で握り、一次承認者のチェック業務を効率化することを目的とする。

- データ取得可否
- 突合キー
- 判定ルール
- 差戻し運用
- 段階的な自動化範囲

### 3.3 Matching Logic（突合ロジック）

- 突合キーは原則 **「社員番号」を主キー**とする。伝票番号や日付等を補助キーに設定。
- 氏名のみの突合は名称揺れ・同姓同名リスクがあるため、社員番号が取得できない場合の補助扱い。
- 社員番号が取得できない可能性があるため、氏名を活用した突合も実施する。突合できず弾かれる内容は、どちらかのデータ表記が間違っているため、社員名称を修正して突合する。

| Matching Target | Primary Key | Supplementary Key / Field | Missing / Multiple Match Rule |
| --- | --- | --- | --- |
| 楽々勤怠 → 楽々精算 | 社員番号 + 出張日 / 勤務日 | 移動開始・終了時刻、勤務開始・終了時刻、勤怠区分 | 勤怠未入力または日付不一致＝「未突合」。同一日に複数勤怠候補＝「複数候補」。 |
| 領収書 / 添付 → 楽々精算 | 申請番号 + 明細番号 + 添付ID | 領収書有無、金額、取引日、取引先名・税金（宿泊税・入湯税） | Phase1は添付有無・金額一致・税金金額を中心に判定。OCR読取は後続フェーズ。 |
| 規程マスタ → 楽々精算 | 出張区分 + 距離条件 + 宿泊有無 + 日数 | 日当、滞在補助費、移動手当、宿泊費上限、食事補助 | 条件不一致＝「要確認」または「NG」。規程の例外は人が判断できるよう理由を出力。 |
| 承認ルート → 楽々精算 | 所属 + 申請者 + 承認者 | 一次承認者、二次承認者、代理承認者 | 想定ルート外＝「承認ルート要確認」。代理承認の許容条件をマスタで明示。 |
| 楽々精算内の重複確認 | 社員番号 + 出張日 + 訪問先 + 金額 | 申請番号、伝票番号、明細番号 | 同一条件で複数申請＝「二重申請疑い」。片方が取消・差戻し済みは除外条件を定義。 |

### 3.4 Validation Rules（検証ルール）

| Category | Fields | Validation Rule | Judgement |
| --- | --- | --- | --- |
| 出張実態 | 出張期間、出張先、目的 | **OK**：対象日の勤怠が入力・確定し、勤務（出勤）扱いが確認できる／**要注意**：勤怠入力済だが未確定・一部欠落（打刻漏れ等）で出勤実態が断定できない／**NG**：勤怠未入力・欠勤・終日休暇等で出勤実態が確認できない（精算前提が崩れる） | OK / 要確認 / 未突合 |
| 労務・健康管理 | 移動開始/終了、勤務開始/終了、深夜・休日・長時間、連続出張 | **OK**：出発／帰着ともに勤怠と精算が一致／**要注意**：差異ありだが軽微（数十分程度・入力ミス濃厚・コメントで理由明確）または修正依頼で整合可能／**NG**：差異が大きい・複数箇所で矛盾・理由が不明／不自然・勤怠未入力で照合不可 | OK / 注意 / 要確認 |
| 出張費・宿泊費上限 | 交通費、宿泊費、日当、滞在補助費、食事補助、手当計、宿泊数 | **OK**：規程の上限・条件に収まる（区分・等級も適正）／**要注意**：上限超過の可能性あるが例外規定対象で根拠（理由・承認）添付／記載あり、または判定情報（区分・泊数・場所）不足で確認待ち／**NG**：上限超過が明確で例外根拠なし・規程上認められない支出 | OK / 要確認 / NG |
| 二重申請・過払い | 社員番号、出張日、訪問先、金額、申請番号、マイナス精算 | **OK**：重複なし／**要注意**：類似申請あり（同日・同額・同区間など）→差分説明が必要／**NG**：同一領収書／同一費用の重複が確定（過払いリスク確定） | OK / 要確認 / NG |
| 証跡・領収書 | 領収書添付、有無、取引日、取引先名、金額、宿泊税/入湯税 | **OK**：領収書と申請内容が一致（税も矛盾なし）／**要注意**：軽微な不一致・確認不足（表記ゆれ、税内訳不明、領収書不鮮明）／**NG**：金額不一致、宿泊先不一致、税の二重計上、領収書と異なる計上 等の不整合が明確 | OK / 要確認 / NG |
| 承認ルート | 所属、承認者、代理承認者、承認ステータス | 本来の承認者が確認すべき申請か、代理承認条件を満たすか | OK / 要確認 |

> ※「軽微／大きい」の閾値（例：30分・60分・90分）は会社運用で決め打ちした方がブレない。必要なら閾値案も作成可能。

## 4. Known Data Issues（既知のデータ課題）

1. システムごとに出力列名・日付形式・時刻形式が異なる可能性。例：`2026/05/20`, `2026-05-20`, `5/20`, 時刻のみ等。
2. 楽々勤怠の社員番号は**正式な社員番号とツール番号の2つ**が活用されている。7月以降に全社員社員番号へ修正予定だが、現状は数人のみ社員番号を活用。
3. 同一社員が同日に複数の出張・外出・作業を行う場合、社員番号×日付だけでは一意に突合できない。
4. 訪問先・取引先名に表記揺れ。例：`A社`, `株式会社A`, `A社 東京事業所`。
5. 領収書添付は画像・PDF・複数枚など形式が分かれる。OCRは初期スコープ外、添付有無・金額確認を優先。
6. 勤怠は出張者がまとめて入力することがあり、勤怠未入力・後日修正・代理入力の扱いを明確化する必要。
7. 規程判定には50km・100km条件、通勤経路外、時間外移動、移動手当、食事補助、宿泊税・入湯税等の細かい条件が含まれる。

## 5. Required Processing Steps（処理ステップ）

> 初期構築はチェックシート作成をファーストアクションとする。Phase1では「自動判定結果を承認者が目検し、必要に応じて手動で差戻し指示する」状態を作る。

| Step | Action | Detail |
| --- | --- | --- |
| Step 0 | 前提確認 | 対象期間、対象部門、対象申請、取得可能データ、取得担当、保存フォルダ、ファイル命名規則を決める。 |
| Step 1 | 判定ルール定義 | 一次・二次承認者の確認観点、規程条件、OK/注意/要確認/NG、差戻し文面候補をマスタ化。 |
| Step 2 | データ取得・格納 | 楽々精算、楽々勤怠、Salesforce、マスタ類を指定フォルダへ格納。初期は手動DL、将来はRPA化を検討。 |
| Step 3 | データ読込・標準化 | 列名、日付、時刻、社員番号、金額、訪問先名を標準化。必須列不足・ファイル不足は取込ログに出す。 |
| Step 4 | 名寄せ・突合 | 社員番号×日付を主キーに楽々精算・勤怠・Salesforceを突合。訪問先、WO、申請番号を補助キーに。 |
| Step 5 | 判定処理 | 出張実態、労務・健康、金額規程、二重申請、領収書、承認ルートを判定し、判定理由を付与。 |
| Step 6 | チェックシート生成 | 一次承認チェック、二次承認詳細、差異一覧、差戻し文面候補、取込ログを1つのExcelに出力。 |
| Step 7 | 承認者レビュー | 承認者が目検。OKは承認、要確認/NGは本人へメールまたは直接差戻し指示。 |
| Step 8 | 修正後再確認 | 修正された申請を再度チェックシートに取り込み、差分が解消されたか確認。 |
| Step 9 | 自動化拡張 | Phase2で差戻しメールのボタン送信、Phase3でチェック→差戻しまでの自動化を検討。 |

## 9. Business Logic Notes（業務ロジック）

### 9.1 Judgement Definitions（判定定義）

| Judgement | Meaning | Action |
| --- | --- | --- |
| OK | ルール上問題なし、または突合結果に差異なし。 | 原則承認可。 |
| 要確認 | 承認判断前に確認が必要。例：勤怠未入力、金額上限付近、備考不足。 | 本人または関係者へ確認。 |
| NG | 規程違反または証跡不足の可能性が高く、そのまま承認すべきでない状態。 | 差戻し、修正依頼。 |
| 未突合 | 必要な相手データが見つからない状態。 | データ未取得・入力漏れ・キー不一致を確認。 |

### 9.2 Phase Approach（フェーズ方針）

| Phase | Automation Level | Operation |
| --- | --- | --- |
| Phase1 | チェックシート自動作成 + 人による判断 | データを手動DLし、Excel/VBAまたはPythonでチェックシートを作成。差分表示後、承認者が目検し、メールまたは直接差戻し指示。チェックシートで承認・差戻を楽々精算に反映できること。 |
| Phase2 | 差戻し連絡の一部自動化 | Power Automate等で、要確認・NGの内容をボタン1つで本人へメール送信。RPAでDL・格納・メール送信補助を検討。 |
| Phase3 | チェックから差戻しまで自動化 | 判定結果に基づき差戻し対象を自動通知。承認判断の完全自動化ではなく、承認者の最終判断を残す。 |

### 9.3 Role Split（役割分担）

- **一次承認者**：出張実態、労務・健康管理、出張先・目的、勤怠との整合、本人への差戻しを主に確認。
- **二次承認者 / 管理部門**：領収書、金額、規程、宿泊費上限、税・手当、過払い・二重申請を主に確認。
- **FPT**：データ構造確認、判定ルール定義支援、チェックシート設計、VBA/Python/RPA/Power Automateの実装・テストを支援。
- **業務責任者**：規程解釈、承認者の判断範囲、要確認・NG条件、例外承認の運用を決定。

## 10. Constraints（制約）

1. **Power Queryは初期実装では前提にしない**。Excel VBA、Python、RPA、Power Automateを中心に検討。
2. 初期段階では、システムからの自動DL・API連携・自動承認は行わない。手動DLしたファイルを指定フォルダに置く方式が前提。
3. **元データは変更しない**。チェック結果は別ファイルとして出力する。
4. 未突合・複数候補・例外条件は削除せず、承認者が判断できる状態で出力する。
5. 楽々勤怠の完全自動化は難しい可能性があるため、勤怠未入力・修正漏れは本人確認または手動差戻しを残す。
6. 領収書OCRは初期スコープ外。Phase1では添付有無・金額・取引日の確認を中心にする。
7. 個人情報・勤務情報・精算情報を扱うため、アクセス権限、保存先、ファイル共有範囲、ログ管理を事前に決める。
8. チェック結果は承認判断を補助するものであり、最終承認・差戻し判断は承認者が行う。

## Appendix B. First Action（最初の着手）

まず着手する内容は**チェックシートの作成**。いきなり全自動化を目指すのではなく、氷山の一角として見えている承認チェック業務を具体化し、動かしながら不足データ・未定義ルール・例外処理を洗い出す。

1. 楽々精算の未承認一覧、楽々勤怠のサンプルデータを1週間分・1か月分取得する。（どれくらい必要？）
2. 一次承認者・二次承認者が現在見ている確認観点を、チェック項目として一覧化する。
3. 社員番号を主キーとして、突合できる項目・できない項目を検証する。必要に応じて補助キーを設定する。
4. OK/注意/要確認/NGの暫定判定ルールを設定し、チェックシートを試作する。
5. 承認者に試作版を見せ、「これなら効果が出そうか」「判断に足りない項目は何か」を確認する。
6. 不足データ、追加ルール、例外条件を反映し、Phase1の運用に入る。
