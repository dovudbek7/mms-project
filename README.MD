# ソースコード構成説明書

出張精算 承認チェックシート 自動生成システム — Phase 1

---

## 1. システム概要

本システムは、出張精算申請の承認業務を補助するPythonプログラムです。

**従来の課題：** 承認者が「楽々精算（出張精算）」と「楽々勤怠（出勤簿）」を別々の画面で手動照合する必要があり、確認漏れや返戻基準のばらつきが発生していた。

**本システムの解決策：** 2つのシステムのデータを自動照合し、承認者が1行で判断できる「統合チェックシート（Excel 7シート）」を自動生成する。

| Phase | 内容 |
|---|---|
| **Phase 1（現在）** | チェックシートの自動生成。最終判断は承認者が行う |
| Phase 2 | 返戻メール自動送信（Power Automate連携） |
| Phase 3 | 返戻処理の自動化（最終承認判断は引き続き人が行う） |

## 4. 判定ステータス語彙

| ステータス | 意味 | セル色 | 承認者の対応 |
|---|---|---|---|
| **OK** | 問題なし | 緑 | そのまま承認可 |
| **要確認** | 承認前に確認が必要 | 黄 | 確認後に判断 |
| **NG** | 規程違反・証憑不足・未承認 | 赤 | 返戻 |
| **未突合** | 対応するマスタデータなし | グレー | データ/キー確認 |
| **複数候補** | 突合候補が複数で1件に確定不可 | 赤 | マスタ確認 |
| **未確認（勤怠データ欠落）** | 対象月の勤怠が未提供 | グレー | 勤怠データ提供後に再判定 |
| **未確認（規程未提供）** | 旅費規程未提供で上限判定不可 | グレー | 規程提供後に再判定 |

**総合判定** = 6軸の中で最も重大なステータス（NG > 要確認 > 未突合 > 未確認 > OK）

## 5. 出力Excel 7シート仕様

| シート名 | 対象者 | 内容 |
|---|---|---|
| `01_一次承認チェック` | 一次承認者 | 申請1件1行：総合判定・要確認項目・差戻し候補 |
| `02_二次承認詳細` | 二次承認者 | 明細レッグ単位：金額・領収書・宿泊・日当・距離 |
| `03_差異一覧` | 承認者全般 | OK以外の項目のみ：理由・確認先システム・対応案 |
| `04_差戻し文面候補` | 承認者全般 | 申請者への返戻メール文面（テンプレート） |
| `05_取込ログ` | 運用担当 | 取込件数・エラー・未承認件数照合 |
| `06_判定ルール` | 業務責任者 | 使用した閾値・距離区分・規程提供状況 |
| `07_マスタ確認` | 管理者 | 未登録・重複・不整合の社員/顧客 |

### [`src/main.py`](./src/main.py)

**役割：** システムのエントリーポイント。5段階のデータ読み込み → エンリッチ → 評価 → Excel出力 を順次実行するパイプラインを制御する。

**主な関数：**

| 関数 | 説明 |
|---|---|
| `run(cfg, stamp)` | メイン処理フロー。5つのローダー呼び出し → `build_check_sheet()` → `write_excel()` → 結果サマリー表示 |
| `_step(no, label, fn, path)` | ローダーをラップ。読み込み件数・エラー・タイムスタンプを取込ログ用に記録 |
| `main(argv)` | CLIエントリー。`--config`（JSON設定ファイル）・`--stamp`（タイムスタンプ指定）を受け付ける |

**実行方法：**
```bash
pip install -r src/requirements.txt
python3 src/main.py
# または設定ファイル指定
python3 src/main.py --config my_config.json
```

### [`src/config.py`](./src/config.py)

**役割：** すべての設定値・閾値・ファイルパス・エイリアスを一元管理する設定クラス。ルールの変更はこのファイルだけで完結する。

**主な設定項目：**

| 設定 | 内容 | 既定値 |
|---|---|---|
| `place_match_threshold` | 行先ファジーマッチ閾値（rapidfuzz partial_ratio） | `82` |
| `fuzzy_name_threshold` | 氏名ファジーマッチ閾値（difflib ratio） | `0.85` |
| `late_night_start_before` | 深夜判定：この時刻以前の移動開始 | `5`（=05:00未満） |
| `late_night_end_after` | 深夜判定：この時刻以降の移動終了 | `22`（=22:00以降） |
| `name_aliases` | 氏名表記ゆれ対応辞書（例：`"張学シン"→"張学鑫"`） | 設定可能 |
| `place_aliases` | 地名・会社名表記ゆれ対応辞書 | 設定可能 |
| `known_gaps` | データ欠損の既知事項（Excelバナー表示用） | 旅費規程未提供など |
| `amount_limits` | 距離区分別日当・宿泊上限（旅費規程値を入力） | 未設定（規程未提供） |
| `receipt_required_above` | 領収書必須の金額閾値 | 未設定（規程未提供） |

**判定ルール欠如時の動作：**
- `has_amount_rules()` → `False`：金額チェックを `未確認（規程未提供）` として処理（NGにしない）
- `has_receipt_rule()` → `False`：領収書チェックを `未確認（規程未提供）` として処理

### [`src/models.py`](./src/models.py)

**役割：** システム全体で共有するデータ型・ステータス語彙の定義。ローダー・マッチング・ルール・出力の全モジュールがこのファイルを参照する。

**主なデータクラス：**

| クラス | 説明 |
|---|---|
| `ExpenseReport` | 1件の出張精算申請（伝票番号・申請者・経費明細リスト・承認者リスト） |
| `ExpenseLeg` | 経費明細1行（交通手段・出発地・目的地・金額・領収書・手当コード） |
| `AttendanceDay` | 従業員1人の1日の勤怠（出退勤時刻・テレワーク・移動時間・出勤状態） |
| `Employee` | 社員マスタ1件（社員番号・氏名・メールアドレス・部署・管轄） |
| `Customer` | 顧客マスタ1件（顧客番号・名称・距離区分・都道府県・地域） |
| `ApproverRule` | 承認者一覧1件（申請者名 → 出張命令者・勤怠承認者・部署情報） |
| `AttendanceLookup` | 勤怠データへの高速インデックス（正規化氏名＋日付でO(1)アクセス） |

**ステータス重大度順：**

```
NG(4) > NEEDS_CHECK/MULTI(3) > UNMATCHED(2) > ATT_MISSING/RULE_MISSING(1) > OK(0)
```

**重要関数：**
- `worst_status(statuses)` — 複数の判定結果から最も重大なステータスを返す集約関数
- `to_axis_vocab(status, axis)` — 内部ステータスをシート出力用の表示語彙に変換

### [`src/normalize.py`](./src/normalize.py)

**役割：** 氏名・地名・金額・日付・時刻の正規化処理を一元化するユーティリティ。全モジュールがこのファイルの関数のみを使用することで、表記ゆれ対応を一箇所に集約する。

**主な関数：**

| 関数 | 入力例 | 出力例 | 用途 |
|---|---|---|---|
| `norm(s)` | `"土田　光樹"` | `"土田光樹"` | 氏名・全般の正規化（NFKC＋全空白除去） |
| `norm_approver(s)` | `"田中一郎(確認)"` | `("田中一郎", True)` | 承認者名から確認専用フラグを分離 |
| `normalize_place(s)` | `"メイコー石巻"` | `("メイコー", "石巻")` | 地名をベース名＋サイトトークンに分解 |
| `parse_band(s)` | `"150~200 km"` | `(150, 200)` | 距離区分文字列を数値タプルに変換 |
| `parse_money(s)` | `"¥ 12,500"` | `12500` | 金額文字列を整数に変換 |
| `parse_jp_date(s)` | `"2026/05/15"` | `date(2026, 5, 15)` | 日付文字列をdateオブジェクトに変換 |
| `parse_time(s)` | `"22:30:00"` | `time(22, 30, 0)` | 時刻文字列をtimeオブジェクトに変換 |

### [`src/checksheet.py`](./src/checksheet.py)

**役割：** システムの中核評価エンジン。データのエンリッチメント（社員・顧客紐付け）を行い、6軸判定を実行し、7シート分の出力データ辞書を構築する。

**主な関数：**

| 関数 | 説明 |
|---|---|
| `enrich_reports(reports, emp_idx, cust_idx, cfg)` | 申請者名を社員IDに解決。各経費明細の行先を顧客マスタに紐付け、距離区分を付与 |
| `build_check_sheet(reports, att_lookup, emp_idx, approver_idx, cfg)` | 全申請に対して6軸ルールを実行し、7シート用データ辞書を返す |
| `_build_master_check(reports, emps, approver_rules, cfg)` | 07_マスタ確認シート用：未突合申請者・承認者一覧との不整合・重複氏名を検出 |
| `_build_rule_rows(cfg)` | 06_判定ルールシート用：使用した閾値・ルール有無を整形 |

**エンリッチメントの設計：**
- 移動系交通手段（電車・新幹線・バス等）は顧客マッチングをスキップ（交通拠点を顧客として誤認識しないよう設計）
- 行先が顧客マスタに一致した場合のみ距離区分を経費明細に付与

**出力データ構造（7シート対応）：**

| キー | 対応シート |
|---|---|
| `primary` | 01_一次承認チェック |
| `secondary` | 02_二次承認詳細 |
| `diff` | 03_差異一覧 |
| `reject` | 04_差戻し文面候補 |
| `import_log` | 05_取込ログ |
| `rules` | 06_判定ルール |
| `master_check` | 07_マスタ確認 |

## 7. ローダー詳細（`loaders/`）

---

### [`src/loaders/expense_loader.py`](./src/loaders/expense_loader.py)

**役割：** 楽々精算からエクスポートされたCP932エンコードCSVを読み込み、`ExpenseReport`（申請単位）と `ExpenseLeg`（明細行）に変換する。

**対応データ：**
- 伝票番号・申請者名・承認者（最大5段階）・承認日・申請日
- 経費明細：日付・交通手段・出発地・目的地・金額・領収書有無・手当コード（日当/宿泊/滞在費補助）

**重要な設計ポイント：**
- ヘッダー名を部分一致で解決（`"ヘッダ情報:"` プレフィックスや `"(件)"` サフィックスの揺れに対応）
- 承認者欄の `"(確認)"` サフィックスを `norm_approver()` で解析し、確認専用フラグを付与
- 申請単位で伝票番号グループ化 → `ExpenseReport.legs` リストに格納
- 移動系明細（出発地・目的地の両方が存在＋移動系交通手段）を自動識別

---

### [`src/loaders/attendance_loader.py`](./src/loaders/attendance_loader.py)

**役割：** 出勤簿Excelファイル（フレックス/固定×一般/管理職の4シート）を読み込み、`AttendanceDay` オブジェクトに変換する。

**出勤状態の判定ロジック（優先順）：**

| 優先度 | 条件 | 判定結果 |
|---|---|---|
| 1 | 出勤打刻あり（休日カレンダー） | `HOLIDAY_WORK`（休日出勤） |
| 2 | 出勤打刻あり（平日） | `OFFICE`（出勤） |
| 3 | テレワーク打刻あり | `TELEWORK`（テレワーク） |
| 4 | 休日カレンダー＋休日出勤の証拠あり | `HOLIDAY_WORK` |
| 5 | 休日カレンダー＋働いていない | `NONWORK`（休日） |
| 6 | 平日＋休暇申請あり | `LEAVE`（休暇） |
| 7 | 上記以外 | `UNKNOWN` |

**対応する移動時間列：** 移動開始/終了（最大2ペア）

---

### [`src/loaders/employee_loader.py`](./src/loaders/employee_loader.py)

**役割：** パスワード暗号化された社員マスタExcel（`社員リスト_20260608.xlsx`）を復号し、`Employee` オブジェクトリストを返す。

**処理フロー：**
1. `msoffcrypto-tool` でパスワード復号 → BytesIOに展開
2. Sheet1を `openpyxl` でread_onlyモードで開く
3. ヘッダー名の正規化マッチングでカラムインデックスを解決（列順に依存しない）
4. 正規化氏名の重複は先勝ちで排除

---

### [`src/loaders/customer_loader.py`](./src/loaders/customer_loader.py)

**役割：** パスワード暗号化された顧客マスタExcel（`顧客リスト_20260608.xlsx`）を復号し、`Customer` オブジェクトリストを返す。

**処理フロー：**
1. パスワード復号
2. 顧客名を `normalize_place()` で `(ベース名, サイトトークン)` に分解
   - 例：`"メイコー石巻"` → `base="メイコー"`, `site_token="石巻"`
3. 距離区分文字列を `parse_band()` で `(下限km, 上限km)` タプルに変換
   - 例：`"150~200 km"` → `(150, 200)`

---

### [`src/loaders/approver_loader.py`](./src/loaders/approver_loader.py)

**役割：** 承認者一覧Excelの「20期」シートを読み込み、結合セル（部署・課・チーム列）を展開して `ApproverRule` リストを返す。

**重要な処理：**
- 結合セルを列単位で独立展開（B列=部署・C列=課・D列=チームの結合が混在）
- 部署の縦持ちフォワードフィル（上位の部署ラベルを下位行に引き継ぐ）
- 対象範囲：6行目〜63行目（53名分）

## 8. マッチング詳細（`matching/`）

---

### [`src/matching/employee_match.py`](./src/matching/employee_match.py)

**役割：** 申請者名（楽々精算の入力値）を社員マスタの社員IDに紐付ける。

**3段階マッチング：**

```
1. 完全一致
   norm(申請者名) == norm(社員名) → 「突合」

2. エイリアス一致
   name_aliases辞書で別表記変換後に再ルックアップ → 「別名突合」

3. ファジーマッチ
   difflib.SequenceMatcher.ratio() ≥ 0.85 → 「別名突合」

4. すべて失敗 → 「未突合」
```

---

### [`src/matching/approver_match.py`](./src/matching/approver_match.py)

### [`src/matching/place_match.py`](./src/matching/place_match.py)

**役割：** 経費明細の行先テキストを顧客マスタに紐付け、距離区分を解決する。システム中で最も複雑なマッチングロジック。

**2段階マッチング：**

```
1. 完全ベース一致
   normalize_place(行先) のベース名が顧客マスタのベース名と一致

2. ファジーマッチ
   rapidfuzz.partial_ratio(行先, 顧客名) ≥ 82 → 候補化
```

**タイブレーク戦略（優先順）：**

| 優先度 | 条件 | 結果 |
|---|---|---|
| 1 | サイトトークン（`"石巻"`等）が候補を一意に絞り込む | 確定 |
| 2 | クエリが候補名の60%未満の長さ（偽陽性ガード） | 除外 |
| 3 | 顧客番号重複排除後に候補が1件 | 確定 |
| 4 | 全候補が同一距離区分 | 確定 |
| 5 | 上記すべて解決不能 | `MULTI`（複数候補） |

### [`src/rules/trip_reality.py`](./src/rules/trip_reality.py) — ①出張実態

**チェック内容：** 出張申請日に実際に勤務・移動の記録があるか。

**判定フロー：**

```
出張日リストを収集（移動系・到着系の明細が対象）
    │
    ├─ 対象月の勤怠データが欠損 → ATT_MISSING（暫定）
    │
    └─ 勤怠データと照合
           │
           ├─ 休暇日と出張日が重複 → NG
           ├─ 移動時間記録あり / 顧客突合済み → 裏付けあり
           └─ 勤怠記録なし → NEEDS_CHECK
               │
               集約：休暇重複→NG、全行先突合＋裏付け→OK
                      それ以外→NEEDS_CHECK
```

### [`src/rules/labor.py`](./src/rules/labor.py) — ②労務

**チェック内容：** 移動時間が勤怠記録と整合するか。深夜移動に手当が付いているか。

**判定ロジック（明細レッグ単位で評価）：**

| 条件 | 判定 |
|---|---|
| 対象月の勤怠データが欠損 | `ATT_MISSING` |
| 深夜移動（設定時間外）＋深夜手当の記録なし | `NEEDS_CHECK` |
| 勤務日でない日の移動（休暇・非勤務日） | `NEEDS_CHECK` |
| 上記以外 | `OK` |
| 残業・休日限度額（旅費規程未提供） | `RULE_MISSING` 追記 |

### [`src/rules/amount.py`](./src/rules/amount.py) — ③金額・限度額

**チェック内容（2系統並行）：**

**① データ整合性チェック（旅費規程なしでも実行）：**

| 条件 | 判定 |
|---|---|
| 明細合計 > 申告合計（真の超過） | `NG` |
| 明細件数の不一致 | `NEEDS_CHECK` |
| 負の金額・小計不一致 | `NG` |
| 合計 ≠ 明細合計（手当分の差異） | 正常（NGにしない） |

**② 規程準拠チェック（旅費規程提供時のみ）：**

| 条件 | 判定 |
|---|---|
| 距離区分ごとの日当・宿泊上限超過 | `NG` |
| 距離区分が未解決（顧客未突合等） | `NEEDS_CHECK` |
| 旅費規程未提供 | `RULE_MISSING` |

### [`src/rules/receipt.py`](./src/rules/receipt.py) — ④領収書

**チェック内容：** 領収書添付が必要な経費に証憑があるか。

**判定ロジック（明細レッグ単位）：**

| 条件 | 判定 |
|---|---|
| 金額ゼロ | `OK`（除外） |
| 領収書添付あり | `OK` |
| IC・運賃系（免除交通手段）かつ金額が閾値未満 | `OK` |
| IC・運賃系かつ金額が閾値以上 | `NEEDS_CHECK` |
| 免除外・領収書なし（旅費規程あり） | `NEEDS_CHECK` |
| 免除外・領収書なし（旅費規程未提供） | `NEEDS_CHECK`＋`RULE_MISSING` 注記 |

### [`src/rules/duplicate.py`](./src/rules/duplicate.py) — ⑤二重申請

**チェック内容：** 同一人物が同一日・同一ルートで複数申請していないか。

**2段階検知：**

| 種別 | マッチングキー | 判定 |
|---|---|---|
| **強一致**（確定重複） | 申請者＋日付＋出発地＋目的地＋交通手段＋金額 | `NG` |
| **弱一致**（疑い重複） | 申請者＋日付＋目的地（顧客番号 or 正規化名） | `NEEDS_CHECK` |

**除外：** 同一伝票内の往復（正当なラウンドトリップ）は重複として扱わない。

### [`src/rules/approval_route.py`](./src/rules/approval_route.py) — ⑥承認ルート

**チェック内容：** 申請者の出張命令者（承認者一覧の `出張命令者` 列）が実際の承認者リストに含まれているか。

**判定ロジック：**

| 条件 | 判定 |
|---|---|
| 承認者一覧に申請者なし（未突合） | `UNMATCHED` |
| 承認未実施 | `NEEDS_CHECK` |
| 期待する出張命令者が実承認者に含まれる | `OK` |
| 確認専用承認者（`(確認)`）のみの場合 | `NEEDS_CHECK` |
| 期待する承認者が不在 | `NEEDS_CHECK`（実際の承認者名を証拠として記録） |

## 11. 設計上の原則

| 原則 | 実装 |
|---|---|
| **元データ不変** | 入力ファイルは読み込みのみ。出力は別ファイルに書き込む |
| **欠損は可視化** | 未突合・欠損データは削除せず、ステータスとして出力に残す |
| **ルール欠如は透明に** | 旅費規程未提供時はNGにせず `未確認（規程未提供）` として出力 |
| **正規化は単一責任** | `normalize.py` のみが正規化を担当。各モジュールが独自実装しない |
| **重大度集約** | 複数軸の結果は `worst_status()` で最も重大なステータスに集約 |
| **Phase1スコープ** | API連携・自動承認・自動ダウンロードなし。ファイル手動配置→実行→Excel確認 |
