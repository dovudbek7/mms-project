# 出張精算 承認チェックシート 自動生成 (Phase1)

楽々精算(出張精算)と楽々勤怠(出勤簿)+各マスタを突合し, 一次/二次承認者向けの
**統合チェックシート(Excel 7シート)** を自動生成する Python 実装。

承認判断を補助するもので, 最終承認・差戻し判断は承認者が行う(Phase1 方針)。

## 実行方法

```bash
pip install -r src/requirements.txt        # openpyxl, msoffcrypto-tool, rapidfuzz
python3 src/main.py                         # 既定 config で実行
python3 src/main.py --config conf.json      # 設定上書き(任意)
```

出力: `out/出張精算_承認チェックシート_YYYYMMDD_HHMMSS.xlsx`

## 入力データ (config.py で定義)

| 種別 | ファイル | 用途 |
|---|---|---|
| 出張精算CSV | `出張精算_*.csv` (cp932) | 判定対象。75伝票/692明細。1伝票=1出張(複数レッグ)。 |
| 勤怠(出勤簿) | `出勤簿_日別詳細_*.xlsx` ×2 | 出張日の在席/勤務時刻照合。氏名+日付で突合。 |
| 社員マスタ | `社員リスト_20260608.xlsx` (暗号化) | 氏名↔社員番号↔所属↔メール。pw=`peeg0608`。 |
| 顧客マスタ | `顧客リスト_20260608.xlsx` (暗号化) | 到着地↔取引先 照合 + 距離区分(50/100km判定)。 |
| 承認者名簿 | `20期評価者・承認者一覧_*.xlsx` | 申請者→想定出張命令者(承認ルート確認)。 |

## ⚠️ 既知のデータ欠落 (出力にバナー表示)

1. **2026-05 の出勤簿が未提供** — 勤怠データは A=2026-04, B=2026-06 のみ。出張データ(5月)と
   日付が重ならないため, 出張実態・労務の勤怠照合は劣化し `未確認(勤怠データ欠落)` を出す。
   隣接月での暫定照合は `allow_adjacent_month_attendance`(既定 False)で有効化可能(advisory のみ)。
2. **旅費規定(J-4-1 / 判定ルールマスタ)が未提供** — 日当/宿泊/上限の金額判定は config 既定値依存。
   実値(`amount_limits`/`receipt_required_above`)が無い限り NG は出さず `未確認(規程未提供)`。

> 重要: 合計 = 明細金額合計 + 手当(日当/宿泊/滞在費補助)。手当は金額列に現れず規程で金額化されるため,
> 「合計 ≠ 明細合計」は正常(差額=手当相当)であり NG ではない。NG は 明細合計>合計 の真の超過のみ。

## 判定ステータス語彙

| ステータス | 意味 | 色 |
|---|---|---|
| OK | 問題なし | 緑 |
| 要確認 | 承認前に確認要 | 黄 |
| NG | 規程違反/証跡不足/未承認 | 赤 |
| 未突合 | 必要な相手データが見つからない | 灰 |
| 複数候補 | 突合候補が複数で1件に確定できない (§6.1) | 赤 |
| 未確認(勤怠データ欠落) | 対象月の勤怠が未提供 | 灰 |
| 未確認(規程未提供) | 旅費規定が未提供で上限判定不可 | 灰 |

総合判定 = 6観点の worst (NG > 要確認 > 未突合 > 未確認 > OK)。

## 6 判定観点 (rules/)

1. **出張実態** `trip_reality.py` — 出張日の勤怠在席/顧客照合。
2. **労務** `labor.py` — 移動時刻 vs 勤務時間, 深夜移動。
3. **金額規程** `amount.py` — データ整合性(合計超過/行数/負値) + 規程上限(規程未提供時は未確認)。
4. **二重申請** `duplicate.py` — (社員,日付,出発,到着,交通,金額) 完全一致のレッグ重複。
5. **領収書** `receipt.py` — 有償・非免除レッグの証票有無。
6. **承認ルート** `approval_route.py` — 申請者の想定出張命令者が実承認者に含まれるか。

## 出力シート (excel_writer.py) — データ契約 §6 の7シート

`01_一次承認チェック`(申請1件1行: 総合判定/要確認項目/差戻し候補) /
`02_二次承認詳細`(明細レッグ単位: 金額/領収書/宿泊/日当/日程/距離) /
`03_差異一覧`(OK以外の観点のみ: 理由/確認先システム/対応案) /
`04_差戻し文面候補`(要確認・NG の観点ごとに差戻し文面) /
`05_取込ログ`(取込件数/未承認件数照合/エラー) /
`06_判定ルール`(使用閾値/区分/規程提供状況) /
`07_マスタ確認`(未登録/重複/不整合)

各観点の判定値は §3.4 の許容語彙へ写像 (`models.to_axis_vocab`). 規程/勤怠の
データ欠落は 要確認 に潰さず `未確認(...)`(灰) として残し, 一次判断のトリアージを保つ。

## モジュール構成

```
src/
├── config.py        設定(パス, pw, 閾値, 規程デフォルト, 既知欠落バナー)
├── normalize.py     正規化(氏名 NFKC+空白除去, 地名, 距離区分, 金額, 日付, 時刻)
├── models.py        dataclass + ステータス語彙 + AttendanceLookup
├── loaders/         expense / attendance / employee / customer / approver 読込
├── matching/        employee(氏名突合+別名) / place(rapidfuzz) / approver
├── rules/           6 判定観点
├── checksheet.py    enrich + 6判定 -> 7シート分の行 dict 組立
├── excel_writer.py  openpyxl で 7シート出力(色分け, バナー, freeze)
└── main.py          オーケストレーション(load→enrich→判定→出力)
```

## 照合ロジックの要点

- **社員番号は系統間キーにならない**(マスタ=AA/AH/AZ, 勤怠A=旧整数ID, 勤怠B=混在)。
  唯一信頼できる横断キーは **norm(氏名)**(NFKC + 全空白除去)。
- 別名辞書 `name_aliases`(例 `張学シン→張学鑫`)で表記揺れを吸収。
- 地名→顧客は normalize_place(株式会社/括弧拠点を除去)+ rapidfuzz partial_ratio(閾値82)。

## 既知の限界 / 次フェーズ

- 5月勤怠を入手すれば出張実態・労務が本判定になる。
- 旅費規定を `config.amount_limits` に投入すれば金額/領収書が本判定になる。
- 領収書OCR・差戻しメール自動送信は Phase2/3。
